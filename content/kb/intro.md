# 소개

주니어 풀스택 엔지니어로서 React + Node.js를 사용하는 플랫폼을 개발하고 있습니다. 우리의 생활에 밀접하고 널리 사용되는 금융, 라이프스타일, 생산성 플랫폼 개발에 직접 참여하는 것이 목표입니다.

## ‘좋은 개발자’라는 꿈

어쩌면 진부한 소재일 수 있는 ‘좋은 개발자’를 늘 소망하고 있습니다.

추상적이고 표현할 수 있는 방법도 많지만, 반드시 대답할 수 있어야 한다고 생각합니다.

저에게는 구체적인 의미로 유연한 개발자입니다.

팀에서 묵묵히 제 역할을 수행하면서도, 또 동료의 의견에 적극적으로 자신의 인사이트를 제시하며 그 유연함을 갖추려고 노력해 왔습니다.

전자는 성실함에서 비롯되지만, 후자는 머리를 쥐어 짜내도 아는 것이 없으면 쉽게 떠오르지 않습니다.

“저도 그렇게 생각합니다.” 외에 의미 있는 대답을 할 수 있도록, 많이 알고 경험하는 것이 제가 추구하는 덕목이 되었습니다.

## 선택의 근거

훌륭한 공학자들이 일궈온 운영체제와 하드웨어들이 그러하듯, 선택에는 항상 트레이드 오프가 있다고 여겨왔습니다.

React 생태계를 흡수해 버린 Next.js 또한 분명히 복잡도 증가와 추가 학습 필요성과 같은 단점이 있음을 이해하였습니다.

이를 의식하며 청크 업로드 테스트 클라이언트와 같은 단순한 SPA 앱은 React만으로 구현함으로써 개발 시간을 아낄 수 있었습니다.

개발을 하다 보면 하나의 기능에도 선택할 수 있는 UI 라이브러리가 다양한 경우가 많았는데, 장단점을 알고 목적이 명확한 접근이 중요했습니다.

예시로 반응형 배너 구현에 `keen-slider` 같은 경량 라이브러리가 아닌 다소 무거운 `swiper`를 선택하였는데, 이는 번들 크기 증가를 감수하는 대신 Parallax와 같은 고급 기능을 활용하여 UX 요구 사항을 충족하고자 하는 목적이 있었습니다.

이처럼 선택의 이유를 스스로에게 납득 시키는 과정이 있기에, 자신만의 Best Practice를 만들어 팀원들의 신뢰를 얻을 수 있었습니다.

## 객체 지향 설계

5원칙의 개념을 익히고 외우는 것 자체는 쉽지만, 정확히 이해하고 실전에 적용하기 위해서는 노하우가 필요하다고 생각했습니다.

그래서 저는 작성한 함수나 모듈의 객체 지향적 요소에 대해서 AI를 활용하여 검증하는 방식을 사용해 왔습니다.

구체적인 예시로, 20개 이상의 입력 필드로 이루어진 콘텐츠 등록 페이지에 사용되는 로직의 복잡도를 고려하여 `useValidator`, `usePopulator` 등으로 모듈화한 후 AI에게 평가를 맡겼었는데, 4개의 모듈이 하나의 Context에 의존하고 있는 문제를 지적해 주었습니다.

이를 받아들여 공통적으로 의존하고 있는 코드를 Hook으로 분리하였고, Context에 대한 모듈의 결합도를 줄이고 재사용성을 개선하여 후에 같은 기능을 React-Admin CMS로 이식하는 데 도움이 되었습니다.

이러한 노하우를 통해 유지 보수가 더 용이한 코드를 작성할 수 있었습니다.
